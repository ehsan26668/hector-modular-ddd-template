# Test Plan: ADR-0027 Domain Event to Integration Event Bridge

## Status

Accepted

## Context

This test plan validates the **Domain Event to Integration Event Bridge** mechanism. As per [ADR-0027](/docs/adr/0027-domain-event-to-integration-event-bridge.md), this bridge is critical for maintaining strict module boundaries in the modular monolith. It ensures internal `DomainEvents` are transformed into stable `IntegrationEvents` within the Application layer and persisted via the `Transactional Outbox` (ADR-0021/0039). This validation is essential to prevent tight coupling between modules and to guarantee reliable, eventually consistent cross-module communication.

## Test Strategy

Define the layers of testing to be used:

- **Unit Tests:**
  - Focus on isolated mapping logic within DomainEventHandlers, correct IIntegrationEventBus usage, and correlation metadata propagation.
  
  - Target Project: `tests/UnitTests/Hector.Modules.Projects.Application.UnitTests`

- **Integration Tests:**
  - Focus on the full transactional pipeline: Domain Event raised â†’ Handler executes â†’ Integration Event persisted in OutboxMessages alongside module state changes.  

  - Target Project: `tests/IntegrationTests/Hector.Modules.Projects.IntegrationTests`

## 1. Scope

- **Included:**
  - Mapping logic from internal Domain Events to external Integration Events.
  - Verification of publishing via `IIntegrationEventBus`.
  - Transactional integrity check (atomic persistence in Outbox).
  - Architectural enforcement (Domain events must not leak outside the module).

- **Excluded:**
  - Transport-specific message broker logic (e.g., RabbitMQ/Kafka configuration).
  - Inbox-side consumer logic (covered in ADR-0023/0035).

---

## 2. Test Cases (Unit / Integration)

- ### TC-01: Should_PublishIntegrationEvent_When_DomainEventHandled

  - **Scenario:** Ensure that a domain event is serialized using its logical contract name and version, not its CLR type name.

  - **Arrange:**
    - Define a domain event with `[OutboxEvent("projects.project-created", 1)]`.
    - Instantiate event with sample data.

  - **Act:**
    - Call `IOutboxEventSerializer.Serialize(event)`.

  - **Assert:**
    - `Type == "projects.project-created"`
    - `Version == 1`
    - `Content` contains valid JSON
    - CLR full type name does NOT appear in serialized metadata.

---

- ### TC-02: Should_MapPropertiesCorrectly_FromDomainToIntegrationEvent

  - **Scenario:** Ensure the data transformation between the internal domain event and the external integration event is accurate.

  - **Arrange:**
    - Define `ProjectCreatedDomainEvent` with specific values (e.g., `ProjectId`, `ProjectName`).

  - **Act:**
    - Trigger handler mapping logic.

  - **Assert:**
    - Verify that properties of the resulting `ProjectCreatedIntegrationEvent` match the domain event fields exactly.

---

- ### TC-03: Should_PersistIntegrationEvent_InOutboxTransactionally

  - **Scenario:** Ensure the Integration Event is stored in the OutboxMessages table within the same database transaction as the primary command execution.

  - **Arrange:**
    - Use `ProjectsDbContext` and a scoped transaction.
    - Execute `CreateProjectCommand`.

  - **Act:**
    - Commit the transaction.

  - **Assert:**
    - Verify database state shows the `Project` entity exists.
    - Verify `OutboxMessages` contains one record containing the serialized `ProjectCreatedIntegrationEvent`.

---

## 3. Non-Functional Validation Points

### 3.1 Security & Sanitization

- Verify that integration event contracts exclude sensitive internal domain fields (e.g., private aggregate state or internal database-only keys) that should not be exposed to other modules.

### 3.2 Observability & Traceability

- Verify that the `CorrelationId` generated by the initial command is correctly propagated from the `DomainEvent` through to the `IntegrationEvent` headers.

### 3.3 Contract Stability

- Verify that integration events reside in the moduleâ€™s `Contracts` assembly, strictly preventing any domain-internal logic from leaking across module boundaries.

## 4. Test Data

- **Inputs:**
  - `ProjectCreatedDomainEvent` (e.g., `ProjectId: Guid.NewGuid()`, `Name: "Test Module Project"`)

- **Expected Outputs:**
  - A serialized JSON record in `OutboxMessages` matching the `ProjectCreatedIntegrationEvent` schema, with `ProcessedOn` as null.

## 5. TDD Execution Plan

Outline the steps for Red-Green-Refactor:

1. **RED:**  
   - Write a unit test that expects an integration event to be published when the domain event handler runs; it should fail as the bridge is not yet implemented.

2. **GREEN:**  
   - Implement the mapping logic within the `ProjectCreatedDomainEventHandler` and call `IIntegrationEventBus.PublishAsync()`.

3. **REFACTOR:**  
   - Move complex mapping logic to a private helper or mapper class if necessary, and verify DI registration for the handler.

## Summary

This test plan ensures that our module boundary remains strictly enforced through [ADR-0027](/docs/adr/0027-domain-event-to-integration-event-bridge.md). By validating the Domain-to-Integration event bridge, we guarantee that other modules only interact with published contracts, preserving the autonomy and integrity of the `Projects` module.
