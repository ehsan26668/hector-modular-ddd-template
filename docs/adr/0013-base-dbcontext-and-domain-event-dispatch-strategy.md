# ADR 0013: Base DbContext and Domain Event Dispatch Strategy

## Status

Accepted

## Context

The project follows a modular monolith architecture with Domain-Driven Design tactical patterns implemented in the Building Blocks layer.

Domain entities derive from `AggregateRoot` and can produce domain events. These events must be dispatched reliably when changes are persisted.

Currently, persistence capabilities exist through EF Core conventions (ADR-0012), including automatic mapping for strongly typed identifiers. However, the system lacks a standardized strategy for:

- Persisting aggregates across modules
- Coordinating transactions
- Dispatching domain events generated by aggregates
- Ensuring consistent behavior across all DbContexts

Without a centralized persistence strategy, each module might implement its own event dispatch logic or transaction handling, leading to duplicated infrastructure code and inconsistent behavior.

To maintain architectural consistency and reduce boilerplate, the system requires a shared persistence base that integrates EF Core with domain event dispatching.

## Decision

We will introduce a shared base class named `HectorDbContext` in the `Hector.BuildingBlocks.Persistence` project.

This base class will extend `DbContext` and provide standardized infrastructure behavior for all module DbContexts.

Responsibilities of the base context:

1. Provide a consistent Unit of Work implementation using EF Core `DbContext`.
2. Automatically collect domain events from aggregates during persistence.
3. Dispatch collected domain events after a successful save operation.
4. Register persistence conventions defined in Building Blocks, such as strongly typed ID mapping.

All module-specific DbContexts must inherit from `HectorDbContext`.

Domain event dispatching will occur during the save pipeline.

Example conceptual flow:

    SaveChangesAsync()
        ↓
    Collect domain events from tracked aggregates
        ↓
    Persist changes to the database
        ↓
    Dispatch domain events via IDomainEventDispatcher
        ↓
    Clear domain events from aggregates

The following infrastructure abstractions will be introduced:

    IDomainEventDispatcher
    HectorDbContext

Example conceptual usage:

    public class SalesDbContext : HectorDbContext
    {
        public SalesDbContext(DbContextOptions<SalesDbContext> options)
            : base(options)
        {
        }
    }

Aggregates continue to raise domain events through the `AggregateRoot` base class, while persistence infrastructure is responsible for dispatching them.

## Consequences

Positive:

- Provides a single, consistent persistence strategy across all modules.
- Ensures domain events are reliably dispatched during persistence.
- Reduces infrastructure duplication across module DbContexts.
- Keeps domain logic free from EF Core dependencies.
- Maintains a clean separation between Domain and Infrastructure layers.

Negative:

- Introduces a dependency on EF Core in the persistence building block.
- Adds additional infrastructure complexity in the base DbContext.
- Requires module DbContexts to inherit from the shared base class.
